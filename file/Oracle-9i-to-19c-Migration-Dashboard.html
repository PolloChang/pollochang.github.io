整體升級流程：9i 到 19c 遷移路線圖 (GoldenGate 近乎零停機方案)

此流程圖展示了如何利用 Oracle GoldenGate (OGG) 的「異質版本即時同步」優勢，將資料從過時的 9i (Source) 遷移至全新的 19c (Target)。請點擊下方任一步驟，查看該階段的詳細營運任務、指令與潛在風險。

1 階段 1：評估與規劃 (Assessment & Planning)

前置準備工作

營運查檢清單 (Operational Checklist)

  • **資料型態評估**：9i 內是否含有被 19c 廢棄或變更行為的欄位（如大容量 `LONG` 欄位需評估轉為 `CLOB`）。
  • **OGG 版本挑選**：GoldenGate 19c 不能直接與 9i 資料庫對接。必須在 9i 安裝較早版本（如 OGG 11g/12c），並在中介伺服器或 19c 端部署 OGG 19c 進行 Trail 格式對應轉換。
  • **字元集（Character Set）確認**：務必確保 9i 與 19c 的 `NLS_CHARACTERSET` 一致（如皆為 `AL32UTF8` 或 `ZHT16MSWIN950`），避免轉換出現亂碼。

宗師提示與避坑指南

在 9i 中，很多 AP 開發人員習慣將密碼與帳號設為相同。**19c 預設啟動密碼區分大小寫且安全性原則嚴格**，請務必先清查所有連線使用者帳號，以免移轉後 AP 因密碼錯誤（ORA-01017）全面中斷。

查詢 9i 目前字元集 SQL：
SELECT parameter, value FROM nls_database_parameters WHERE parameter='NLS_CHARACTERSET';

2 階段 2：19c 環境建置 (19c Target Setup)

目標端配置

營運查檢清單 (Operational Checklist)

  • **安裝空的 19c 資料庫**：建議採用多租戶架構（Container Database, CDB / Pluggable Database, PDB），為未來雲端化做準備。
  • **建立對應的 Tablespace**：在 19c 中建立與 9i 名稱相同、容量充足的資料表空間，避免匯入時因空間不足中斷。
  • **網路通道設定**：防火牆必須開啟 9i 與 19c 主機之間的 **TCP Port 1521** (SQL*Net) 與 **TCP Port 7809** (OGG Manager 專用通訊埠)。

建議的 19c 表空間建立指令

-- 在 19c PDB 容器中建立高彈性 Local Managed 表空間

CREATE TABLESPACE APP_DATA DATAFILE '/u01/app/oracle/oradata/ORCL/pdb1/app_data01.dbf' SIZE 10G AUTOEXTEND ON NEXT 1G MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL SEGMENT SPACE MANAGEMENT AUTO;

3 階段 3：初始化資料載入 (Initial Load & Data Seeding)

基底資料同步

營運查檢清單 (Operational Checklist)

  • **啟動 9i OGG 擷取行程 (Extract)**：先將 9i 的 OGG Extract 啟動，用以捕捉資料初始化載入期間產生的所有即時 DML 變更（防止資料遺漏）。
  • **利用 9i 傳統 Export 匯出**：因為 9i 無 Data Pump，必須使用舊版 `exp` 指令進行 schema 級別匯出，**並務必在指令中加入 `CONSISTENT=Y` 確保匯出資料的時間點一致性**。
  • **記錄匯出時的 SCN (System Change Number)**：此 SCN 為後續 OGG 開始寫入的「起始基準點」。
  • **使用 19c Impdp 匯入**：利用 19c 的 `impdp` 指令將 `.dmp` 檔案匯入 19c 目標庫中。

9i 端傳統匯出 (Export) 指令

exp system/passwd owner=ap_user file=exp_9i_ap.dmp log=exp_9i_ap.log consistent=y

19c 端 Data Pump 匯入 (Import) 指令

impdp system/passwd directory=DATA_PUMP_DIR dumpfile=exp_9i_ap.dmp logfile=imp_19c_ap.log remap_schema=ap_user:ap_user

4 階段 4：即時變更追趕 (Delta Synchronization)

雙軌同步追趕

營運查檢清單 (Operational Checklist)

  • **啟動 19c 傳輸寫入端 (Replicat)**：配置 19c 端 OGG Replicat 參數，**指定起始點為階段 3 所記錄之 SCN 之後的交易**。
  • **監控即時同步延遲時間 (Lag Time)**：使用 OGG 監控工具定期檢查是否有長期延遲或阻塞。
  • **進行資料完整性比對**：在背景同步期間，可執行 OGG Veridata 或撰寫比對 Script，抽樣比對 9i 與 19c 資料列數與欄位雜湊值是否完全吻合。

OGG 營運監控核心命令 (GGSCI)

-- 登入 OGG 控制台並檢查所有行程狀態

GGSCI> info all

-- 檢查 19c 寫入端是否含有延遲（Lag）

GGSCI> info replicat rep_name, showch

5 階段 5：切換 (Switchover - Go Live!)

停機與上線切換

營運切換當日劇本 (Cutover Runbook)

  1. **停止 AP 寫入**：停止所有連結到 9i 的應用系統，以防產生新資料。
  2. **將 9i 設定為唯讀 (ReadOnly)**：預防性將 9i 資料庫鎖定，防止不慎寫入。
  3. **確認 OGG 延遲歸零**：等待 OGG 的 Lag 降為 0，確保最後一筆變更已寫入 19c。
  4. **停止 OGG 所有服務**：完成歷史階段使命，關閉 Extract 與 Replicat。
  5. **修改 AP 連線設定**：更換連線字串（如改為 19c SCAN IP 或更新 DNS 設定）。
  6. **重啟 AP 服務並進行全功能驗證**。

應急回滾（Rollback）方案

安全退路：如果新庫 19c 發生非預期的效能崩潰或相容性問題，因 9i 在切換前狀態被完整保留，且未遭受損壞，營運團隊只需將 AP 連線字串改回 9i 即可立即恢復生產，專案風險極低！

注意：若 19c 已經運行一段時間並產生了大量新交易，若要回滾，需在 19c 側架設反向 OGG Extract 同步回 9i。

Database Role Mapping (Oracle 9i ➡️ 19c 權限修正地圖)

升級後最令營運團隊頭痛的是：「為什麼以前在 9i 跑得好好的 AP 帳號，移到 19c 就狂報 ORA 權限不足錯誤？」
這是因為 Oracle 在 10g/11g/12c 以降大幅收緊了內建角色的權限。以下為您整理舊新版本行為變更，並附帶直接可在 19c 執行的權限修復指令 (Remap SQL)。

CONNECT 角色收緊 高影響度
Oracle 9i 舊版行為
內建 `CONNECT` 角色權限廣泛。只要獲得此角色，便會自動被授予：
  • CREATE SESSION
  • CREATE TABLE
  • CREATE VIEW
  • CREATE SYNONYM 等...
Oracle 19c 新版行為
為了遵循安全最小化原則，在 19c 中 `CONNECT` 被削弱。**目前僅剩下「CREATE SESSION」權限**。
🔴 影響：AP 無法自動建立暫存表或視圖 (ORA-01031: insufficient privileges)
19c 營運修復指令 (Remap SQL)： 
-- 1. 建立自訂的角色，補足 19c 被收回的常用開發權限
CREATE ROLE AP_DEV_ROLE;
GRANT CREATE TABLE, CREATE VIEW, CREATE SYNONYM, CREATE SEQUENCE TO AP_DEV_ROLE;

-- 2. 將此新自訂角色授予您的 19c AP 帳號
GRANT CONNECT, AP_DEV_ROLE TO your_ap_user;
RESOURCE 空間限制 高影響度
Oracle 9i 舊版行為
當用戶被授予 `RESOURCE` 角色時，Oracle 9i 會在幕後自動對該用戶贈送 `UNLIMITED TABLESPACE` 系統權限。
Oracle 19c 新版行為
基於多租戶資源隔離要求，授予 `RESOURCE` 角色**不再附贈 UNLIMITED TABLESPACE 系統權限**。
🔴 影響：AP 寫入大筆資料時會隨時斷線 (ORA-01950: no privileges on tablespace)
19c 營運修復指令 (Remap SQL)： 
-- 必須在 19c 手動顯式指定使用者的表空間配額限制
ALTER USER your_ap_user QUOTA UNLIMITED ON users;

-- 或者若有指定專屬資料表空間：
ALTER USER your_ap_user QUOTA UNLIMITED ON APP_DATA;
